# Specification: Lab Order Requests

## ADDED Requirements

### Requirement: Lab Order As A Recommendation Payload Type

The system SHALL represent lab orders as `Recommendation` rows with
`payload_type = lab_order` and a structured payload containing: panel
(coded set of analytes), indication (free text), fasting required flag,
and acceptable fulfillment paths (Labcorp / Quest / Getlabs). Lab orders
SHALL flow through the same physician-in-loop state machine as every
other recommendation type.

#### Scenario: A lab order draft is persisted with a structured payload

- **GIVEN** a protocol produces a lab order draft for HbA1c
- **WHEN** the draft is persisted
- **THEN** a `Recommendation` row is created with
  `payload_type = lab_order`
- **AND** the payload contains a panel, an indication, a fasting flag,
  and a non-empty list of acceptable fulfillment paths
- **AND** the row's state is `PENDING_REVIEW`

#### Scenario: A lab order is delivered only after physician approval

- **GIVEN** a lab order `Recommendation` in `PENDING_REVIEW`
- **WHEN** the physician approves it
- **THEN** the state transitions to `APPROVED` and then `DELIVERED`
- **AND** the lab order becomes visible to the patient
- **AND** an audit event records the approval

---

### Requirement: Patient-Facing Lab Order Action Card

A delivered lab order SHALL render in the patient app as an action card
containing the panel name, the indication, the fasting requirement, and
a list of acceptable fulfillment paths. The card SHALL allow the patient
to view a printable PDF requisition generated by the backend.

#### Scenario: A patient sees their lab order

- **GIVEN** a `DELIVERED` lab order `Recommendation` for a patient
- **WHEN** the patient opens the conversation containing it
- **THEN** an action card renders showing the panel, indication, fasting
  flag, and fulfillment paths
- **AND** a "View requisition" control returns a backend-generated PDF

#### Scenario: A non-delivered lab order is not visible to the patient

- **GIVEN** a lab order `Recommendation` in `PENDING_REVIEW`,
  `APPROVED`, or `REJECTED`
- **WHEN** the patient lists their visible recommendations
- **THEN** the non-delivered lab order is not included

---

### Requirement: Manual Fulfillment Status Tracking

The system SHALL track lab fulfillment status on a
`lab_order_fulfillments` table keyed by recommendation id. In Phase 1,
status transitions (`pending → scheduled → collected → resulted →
cancelled`) SHALL be applied manually by a clinical operator via an
administrative endpoint; integration with external lab vendors is out of
scope for this change.

#### Scenario: A clinical operator advances a lab order status

- **GIVEN** a `lab_order_fulfillments` row in `pending`
- **WHEN** a clinical operator advances the status to `scheduled` via
  the administrative endpoint
- **THEN** the row's status is updated
- **AND** the patient action card reflects the new status
- **AND** an audit event records the status change

#### Scenario: An invalid status transition is rejected

- **GIVEN** a `lab_order_fulfillments` row in `resulted`
- **WHEN** an operator attempts to transition it back to `pending`
- **THEN** the request is rejected with a typed transition error
- **AND** the row's status is unchanged
